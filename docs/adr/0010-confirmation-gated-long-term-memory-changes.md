# Confirmation-gated long-term memory changes

PaiCLI will continue to create long-term memories only through `/save` or an explicit user request. It will use keyword retrieval followed by an LLM to classify a related existing memory as `duplicate`, `merge`, `replace`, or `independent`; a failed classification is treated as `independent`. Independent facts are saved immediately, and duplicates return the existing memory without a new write. Merge, replacement, or retirement outcomes are stored separately in `~/.paicli/memory/pending_changes.json` and do not participate in retrieval. Each proposal identifies its operation, scope and project, target-memory IDs, proposed final content, rationale, source fact, timestamp, and pending status. Users inspect, apply, or discard proposals through `/memory pending`, `/memory apply <id>`, and `/memory reject <id>`; explicit confirmation is required before persistent memory changes. Retirement is proposed only by an explicit save classified as `replace` or `retire`, never by age; existing recency decay remains a retrieval-ranking signal only. This preserves useful maintenance without allowing model inferences to silently alter stable context.
