# Successful work defines replanning progress

PaiCLI will measure plan progress as successfully completed nodes divided by the original plan's node count; failed, blocked, skipped, and pending nodes do not increase progress. When a node fails, the current parallel batch may finish but no new nodes are scheduled. Descendants of failed nodes become `BLOCKED`, while otherwise runnable nodes remain `PENDING` with a `plan_halted` reason. If successful progress is below the configurable threshold, default 50%, PaiCLI may generate at most one replacement plan containing only unfinished work and must submit it for user review. Completed nodes are treated as non-repeatable facts unless the replacement plan explicitly requires compensation or rollback. At or above the threshold, or after the single replacement plan, PaiCLI returns an aggregate partial result covering every plan attempt.
